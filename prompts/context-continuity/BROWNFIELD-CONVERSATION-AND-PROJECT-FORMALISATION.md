# Brownfield Conversation and Project Formalisation

## Mission

Formalise a conversation, codebase, vault, strategy, creative thread or legacy corpus that began without a complete plan but developed durable value.

Do not assume the opening request still defines the work. Do not treat the latest summary as complete history. Do not create a new project merely because the source estate is large.

## Inputs

- complete accessible conversation or source scope;
- repositories, folders, vaults and provider objects;
- active workers and current revisions;
- destination ecosystem or organisational map;
- privacy, legal and no-touch boundaries;
- mutation authority.

## Procedure

### 1. Establish custody

Inventory exact sources, representations, revisions, attachments, generated artifacts, provider objects, missing material and active workers. Distinguish bytes, parsed text, references and semantic identity.

### 2. Reconstruct what the work became

Identify teleology, users, audiences, objects, capabilities, workflows, decisions, corrections, strong artifacts, reusable prompts, commitments, unfinished work, sensitivities and obsolete assumptions.

### 3. Classify the brownfield case

Choose one or more:

- emergent conversation;
- existing project continuation;
- legacy donor;
- multi-root reconciliation;
- migration or refactor;
- unresolved candidate.

### 4. Build value ledgers

Create source, artifact, decision, prompt, entity-candidate, failure-mode and open-loop ledgers. Each record needs source locators, lifecycle, authority status, unique value, related or superseding objects, sensitivity and recommended action.

### 5. Search before placement

Compare the reconstructed work with existing projects, systems, repositories, owners and capabilities. Evaluate continue, merge, split, donor/archive, local subproject, quarantine or new-node warrant.

### 6. Select deliverables

Use a deliverable type library and project profile to select the smallest coherent set needed for understanding, building, governing, testing, migration, use and continuity. Brownfield work often requires archaeology, lineage, current-state, migration, supersession and archive deliverables.

### 7. Produce the formalisation packet

- reconstructed project or system definition;
- source and value inventories;
- conflict, duplicate and supersession audit;
- owner and placement docket;
- selected deliverable architecture;
- migration or continuation sequence;
- operating spine;
- context pack and handoff;
- project-management and registry patch candidates;
- receipt and exact resumption.

### 8. Migrate carefully

Use bounded owner-local changes, preserve sources and rejected variants, respect active locks, validate destination state and avoid flag-day rewrites.

## Quality gates

- every substantial conclusion resolves to evidence;
- source mention is not byte custody;
- generated output is not acceptance;
- similarity is not identity;
- option types are not instantiated entities;
- chat prose is not canon;
- missing sources remain explicit;
- a fresh authorised worker can continue without repeating archaeology.

## Stop conditions

Stop before mutation when owner, authority, source coverage, sensitivity, active workers, destination or rollback are unresolved. Return a review docket and exact information needed to proceed.

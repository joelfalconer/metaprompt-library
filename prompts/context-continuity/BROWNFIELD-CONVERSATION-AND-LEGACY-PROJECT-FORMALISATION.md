# Brownfield Conversation and Legacy Project Formalisation

```text
Treat the supplied conversation, project, repository, vault, folder tree or package as a living or historical source estate, not automatically as a clean project and not automatically as its future canonical owner.

MISSION
Recover its durable value, reconstruct what it became, determine where it belongs, and establish a safe continuation or migration path without erasing provenance, forks, rejected variants or active work.

1. Establish custody. Inventory conversations, transcript snapshots, attachments, generated artifacts, repositories, branches, vaults, provider records, external outputs, active workers, missing sources and inaccessible references. Distinguish exact bytes, parsed text, representation and semantic identity.

2. Classify the entry:
- emergent conversation;
- valid existing project continuation;
- legacy donor;
- multi-root reconciliation;
- refactor or migration;
- unresolved candidate.

3. Reconstruct what the work became. Identify purpose, outcomes, users, objects, capabilities, workflows, decisions, corrections, strong deliverables, reusable methods, active commitments, sensitivity boundaries, obsolete assumptions and unresolved contradictions. Do not let the opening prompt define the later project.

4. Search existing projects, registries and owners before creating a new node. Test: continue existing owner, merge, split by ownership, retain as donor, create a local subproject, quarantine, or warrant a new node.

5. Build a value ledger. For each significant object record source locator, type, lifecycle, owner candidate, relations, unique value, sensitivity and disposition: preserve, continue, merge, migrate, extract, register, archive, reject or investigate.

6. Select the project-specific deliverable architecture. Include archaeology, lineage, migration, current-state and decision outputs where needed. Avoid decorative documentation and duplicate sources of truth.

7. Produce a placement docket and bounded migration or continuation sequence with owners, destinations, active locks, rollback, verification and unresolved operator decisions.

8. Establish or repair the operating spine: entry guide, boundary, local instructions, current state, work and dependencies, source and artifact registries, decisions, handoffs, receipts, roadmap and exact resumption. Reuse existing owner conventions.

9. Validate that a fresh authorised worker can find the owner, distinguish source from accepted truth, retrieve key artifacts, understand locks and continue the next action without recreating superseded work.

Return:
- source and representation inventory;
- brownfield classification;
- reconstruction brief;
- value ledger;
- conflict, duplication and supersession report;
- placement docket;
- selected deliverable architecture;
- migration or continuation sequence;
- context pack and handoff;
- registry and project-management patches;
- receipt and exact resumption.

Stop before migration when custody, owner authority, sensitivity, active locks, destination or rollback remains unresolved.
```

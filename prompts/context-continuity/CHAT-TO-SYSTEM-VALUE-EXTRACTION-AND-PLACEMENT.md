# Chat-to-System Value Extraction and Placement

## Mission

Transform the complete accessible conversation and its artifacts into durable system value without treating chat prose as canon.

## Governing transformation

```text
ephemeral conversation
→ preserved source
→ turn-local objects
→ lineage and correction
→ placement decisions
→ bounded owner packets
→ verified promotion
→ reusable handoff
```

## Required phases

1. **Source custody:** inventory transcripts, uploads, generated files, archives and missing sources; preserve immutable representations.
2. **Turn indexing:** identify requests, responses, tool attempts, observed results, decisions, corrections, artifacts, blockers and exact resumption.
3. **Occurrence extraction:** assign stable source locators without changing source bytes.
4. **Reconciliation:** model aliases, contradictions, revisions, supersessions and artifact ancestry; preserve rejected variants.
5. **Placement:** search existing systems, projects and registries before proposing a new node. Decide extend, merge, donor, quarantine or warrant-new.
6. **Owner packets:** partition by sensitivity; include pointers, competing hypotheses, review status and rollback.
7. **Promotion:** apply only explicitly approved owner-local changes through bounded work orders.
8. **Closure:** issue receipt, open-loop ledger, handoff and exact next prompt.

## Non-negotiable rules

- A transcript mention does not prove byte custody.
- A generated file does not prove canonical acceptance.
- A tool call does not prove successful execution.
- Similar titles do not prove identical artifacts.
- Do not invent missing content or silently erase contradictions.
- Do not create a new project merely because the chat has a name.

## Required outputs

`SOURCE_INVENTORY`, `TURN_INDEX`, `OCCURRENCE_LEDGER`, `LINEAGE_REGISTER`, `PLACEMENT_DOCKET`, `OWNER_HANDOFFS`, `OPEN_LOOPS`, `RECEIPT`, and `EXACT_RESUMPTION`.

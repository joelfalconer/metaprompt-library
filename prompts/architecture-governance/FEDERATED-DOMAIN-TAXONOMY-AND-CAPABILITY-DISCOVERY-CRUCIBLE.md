# Federated Domain Taxonomy and Capability-Discovery Crucible

## Mission

Transform an incomplete seed into a comprehensive but non-canonical candidate taxonomy for a system, office, realm, programme or product.

## Required distinctions

```text
interest ≠ domain ≠ object ≠ workflow ≠ capability ≠ implementation ≠ tool
permission ≠ authority ≠ execution ≠ verified outcome
category ≠ agent
semantic owner ≠ executor ≠ custodian ≠ approver ≠ verifier
```

## Phases

1. Ground current mandate, neighbours, existing IDs, sources and no-touch boundaries.
2. Produce an `owns / coordinates / consumes / routes / forbids` matrix.
3. Define ontology primitives and stable candidate IDs.
4. Design orthogonal facets: target, object, scope, horizon, cadence, state, context, value, risk, reversibility, sensitivity, evidence, jurisdiction, owner tuple and maturity.
5. Expand macro-domains and detailed leaves with includes, excludes, workflows, metrics, interfaces and failure modes.
6. Audit duplicates, vendors-as-capabilities, ownerless shared work, role/agent confusion, scope collisions, privacy overreach and premature automation.
7. Test with routine, high-stakes and cross-system compound records.
8. Produce a capability-interest matrix before proposing agent topology.
9. Rule candidate, reviewed or rejected status and define the next bounded artifact.

Use progressive disclosure: executive map first, detailed registry later.

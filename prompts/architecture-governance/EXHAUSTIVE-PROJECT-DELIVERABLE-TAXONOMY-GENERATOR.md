# Exhaustive Project Deliverable Taxonomy Generator

```text
Design a reusable deliverable-type library. Do not select documents for one project and do not imply that every project needs every type.

Distinguish deliverable type, artifact instance, template, schema, registry, catalogue, generated projection and evidence receipt.

Build a nested taxonomy covering orientation, research, strategy, product, requirements, architecture, engineering, data, knowledge, semantics, design, brand, creative work, editorial, marketing, commercial work, programme management, operations, quality, security, privacy, governance, delivery, adoption, lifecycle, agent and instruction systems, infrastructure, vaults, templates, migration and brownfield archaeology.

Add profiles for software applications, packages, platforms, data systems, infrastructure, agent systems, Obsidian vaults, brand programmes, campaigns, creative and worldbuilding projects, publishing, research, institutions, migrations and collections.

For each leaf type provide:
- stable candidate ID;
- label, definition and hierarchy;
- purpose and decisions enabled;
- use when and do not use when;
- likely owner and audience;
- typical inputs and outputs;
- canonicality and lifecycle patterns;
- dependencies and related types;
- validation methods;
- common formats or surfaces;
- applicable project profiles;
- aliases, overlaps and deprecation notes.

Also define facets for selection state, ownership, canonicality, lifecycle, sensitivity, audience, medium, validation and maintenance. Identify near-duplicates, common composites, overproduction risks and wrong-surface anti-patterns.

Return:
A. nested human-readable taxonomy;
B. machine-readable type library;
C. facet model;
D. project-profile matrix;
E. alias, overlap and composite map;
F. coverage audit;
G. extension, versioning and deprecation rules.

Treat the result as an option space, never an automatic bureaucracy generator.
```

# Universal Work Item Launcher

```text
WORK ITEM ID OR CODE:
{{ID_OR_CODE}}

WORK ITEM NAME:
{{NAME_OR_UNKNOWN}}

OPTIONAL OPERATOR NOTE:
{{NOTE_OR_NONE}}

You are entering an existing project, programme or work ecosystem to execute, prepare, review or route the Work Item identified above. The supplied fields are locators and operator intent, not the complete instruction set, proof of current state or mutation authority.

PHASE A — DISCOVER AND RESOLVE
1. Search authoritative files, repositories, issue trackers, provider objects, handoffs and registries for the exact code, name, aliases and superseding records.
2. Identify the source of truth and distinguish it from dashboards, mirrors, chat memory, exports and generated projections.
3. Determine whether the item is current, completed, superseded, duplicated, blocked, stale, already assigned, missing formal definition or not warranted.
4. Resolve purpose, outcome, semantic owner, accountable authority, executor, verifier, current revision, active workers, dependencies, blockers, sources, decisions, sensitivity, permissions, locks, write scope, validation, rollback and done conditions.
5. Treat the operator note as emphasis or constraint. It may narrow work but may not override newer authority, locks or safety boundaries.
6. Stop with a routing docket when identity, authority, context or dependencies remain unresolved.

PHASE B — COMPILE MINIMUM SUFFICIENT CONTEXT
7. State selected and deliberately excluded context. Retrieve only what the bounded outcome requires.
8. Preserve provenance and distinguish source evidence, provider observation, operator report, inference, proposal, rejected variant and unknown.
9. Choose one route: CONTINUE_HERE, RETURN_TO_EXISTING_INSTANCE, START_SPECIALIST_INSTANCE, RUN_IN_LOCAL_HARNESS, OWNER_REVIEW_ONLY, WAIT_FOR_DEPENDENCY, DEFER, RETIRE or NO_WORK_WARRANTED.
10. When another worker or owner has stronger context or authority, prepare a task-specific handoff instead of duplicating the work.

PHASE C — START HOOK
11. Immediately before consequential action, reread the target revision, active locks and current work state.
12. Create or claim one bounded Work Order, ticket or equivalent when required.
13. Use an isolated branch, worktree, draft, transaction or review surface where appropriate.
14. Record start revision, intended mutations, validation, rollback and stop conditions.

PHASE D — EXECUTE CONDITIONALLY
15. Perform only the accepted outcome. Revise the current authoritative object rather than creating a parallel source where possible.
16. Apply the relevant mode:
   - RESEARCH: use authoritative current sources, record evidence cutoff and keep findings separate from policy.
   - ARCHITECTURE: preserve candidate versus accepted status, owners, options, migration impact and decision gates.
   - IMPLEMENTATION: read local instructions and tests, preserve unrelated changes, implement the smallest coherent slice and verify the real result.
   - DOCUMENTATION: identify consumer, owner, canonical location and maintenance trigger; revise instead of duplicate.
   - MIGRATION_OR_REFACTOR: inventory source and target, preserve lineage, use reversible batches and avoid flag-day replacement.
   - PROVIDER_MUTATION: read before write, use stable IDs, mutate the smallest object, reread and record before/after state.
   - PRIVATE_OR_SENSITIVE: minimise payload and preserve owner-specific access and disclosure rules.
   - READ_ONLY_AUDIT: do not expand the audit into target mutation.
   - PUBLIC_REUSABLE_PROMPT_OR_TEMPLATE: remove private context, test transferability, search for overlap and register version and lineage.
   - OWNER_REVIEW: do not apply the proposal; return approve, revise, defer or reject with evidence.
17. Preserve corrections, rejected variants, uncertainty, negative evidence and useful history.
18. Do not mark work complete merely because files or prose were generated.

PHASE E — VALIDATE AND CLOSE
19. Run owner-defined tests, schemas, fixtures, provider rereads and human gates. Distinguish attempted, generated, written, published, applied, observed, verified and accepted.
20. Update only necessary durable surfaces: authoritative artifacts, current work state, decisions, roadmap, handoff, receipt, materially stale mirrors and accepted registries.
21. Reread consequential external writes. Record exact revisions, provider IDs, changed objects, validation, failures, residual risk and rollback.
22. Close as COMPLETE_AND_ACCEPTED, COMPLETE_AWAITING_OWNER_ACCEPTANCE, PARTIAL_WITH_SAFE_RESUMPTION, BLOCKED, SUPERSEDED, ROUTED_TO_EXISTING_OWNER, RETIRED or NO_WORK_WARRANTED.
23. Leave the exact next Work Item, required context, responsible owner and first resumption action.

REQUIRED INITIAL RESPONSE
Return a compact docket containing the resolved item and state, owner and authority, current revision, dependencies and blockers, active workers and locks, selected and excluded context, route, tools and mutation scope, intended surfaces, done conditions, validation, rollback and whether work can begin now. Proceed only when resolved authority permits it.
```
